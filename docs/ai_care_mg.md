# ケアマネAI アプリケーション 要件定義書

## 1. プロジェクト概要

ケアマネジャー（介護支援専門員）の日常業務を AI でサポートする Web アプリケーション。  
特定の介護支援事業所を初期ターゲットとし、まず身近なケアマネさんに使ってもらいながら改善を重ねる。

---

## 2. ターゲットユーザー

| 項目 | 内容 |
|------|------|
| 初期ユーザー | 特定の介護支援事業所のケアマネジャー |
| 想定人数 | 10名以上 |
| 利用環境 | PC ブラウザ（Chrome / Edge 推奨）、一部モバイルブラウザ |

---

## 3. 解決する課題

| 課題 | 現状 | このアプリでの解決 |
|------|------|------------------|
| ケアプラン作成に時間がかかる | 平均2時間/件。白紙から専門的な文章を書く必要がある | 音声入力 → 文字起こし → Gemini ドラフト生成で作成時間を削減 |
| 訪問後の記録が二度手間 | 手書きメモ → 帰社後に PC 入力という二重作業 | 訪問直後に音声入力 → Gemini 要約で記録完了 |
| ナレッジが属人化している | 熟練ケアマネのノウハウが共有されない | 個人・共有ナレッジベースを RAG で活用 |
| メール対応の文章作成が手間 | 関係各所へのメール文を毎回書き起こす | Gemini が返信文のドラフトを提案 |

---

## 4. 機能要件（MVP スコープ）

### 4.1 認証・ユーザー管理

- メールアドレス + パスワードによるログイン
- ユーザーロールは「一般ケアマネ」と「管理者」の2種類
- 管理者はユーザーの招待・停止ができる

### 4.2 音声入力 → 文字起こし → 要約

- ブラウザ上でマイクから音声を録音できる
- Web Speech API による文字起こし（精度が不十分な場合は Gemini API の音声入力へフォールバック）
- 文字起こし結果を Gemini API で要約し、利用者の課題・ニーズを構造化して表示する
- 要約結果はケアマネが編集できる

### 4.3 ケアプランドラフト自動生成

- 利用者の基本情報（年齢・疾患・要介護度）と要約結果を入力として、Gemini API がケアプランのドラフトを生成する
- 生成されるドラフトの構成：長期目標・短期目標・サービス内容の候補
- ケアマネがドラフトを選択・編集して最終プランを完成させる
- 完成したケアプランは PDF または Word 形式でエクスポートできる（**MVP スコープ外**。画面上での閲覧のみ MVP 対応、紙出力は第二フェーズ）

### 4.4 ナレッジベース（RAG）

- **個人ナレッジ**：ケアマネ各自がメモ・ノウハウを登録・管理できる
- **共有ナレッジ**：管理者が介護保険制度資料・マニュアル類を登録し、事業所全員が参照できる
- 登録できるファイル形式：PDF、Word、テキスト
- ケアプランドラフト生成時に RAG としてナレッジを参照する
- ナレッジの追加・削除・更新ができる

### 4.5 メール返信文の提案

- ケアマネがメールの本文をアプリに貼り付ける
- Gemini API が返信文のドラフトを提案する
- ケアマネが確認・編集して、コピーして自分のメールソフトから送信する（アプリからの送信は行わない）

## 5. 機能要件（将来検討）

| 機能 | 理由・条件 |
|------|-----------|
| 音声入力の精度向上（専門用語辞書） | MVP で精度課題が確認されたら対応 |
| モバイルアプリ化（PWA or ネイティブ） | 訪問中の利用ニーズが高まったら検討 |
| 介護ソフトへの自動連携（API 連携） | 連携先ソフトが決まってから設計 |
| 関係各所へのメール一斉送信エージェント | MVP 後の第二フェーズ |
| 介護サービス事業所の空き情報検索 | データソース確保が前提 |
| 給付管理・レセプト連携 | 法規制対応が必要なため慎重に検討 |

---

## 6. 非機能要件

| カテゴリ | 要件 |
|---------|------|
| セキュリティ | 利用者の個人情報を扱うため、通信は HTTPS 必須。Supabase RLS でユーザーごとのデータアクセスを制御する |
| 個人情報保護 | Gemini API に送信するデータは個人を特定できる情報（氏名・住所等）をマスキングする仕組みを設ける（`pii-masking-design.md` で詳細設計済み） |
| パフォーマンス | AI 生成のレスポンスはストリーミング表示で体感速度を改善する（**MVP は非ストリーミングで可**。ストリーミング対応は運用開始後の UX 改善項目。`care-manager-ai-design.md` §10.2 参照） |
| 可用性 | MVP 段階では多少のダウンタイムは許容。Vercel + Supabase のマネージドサービスに依存 |
| ブラウザ対応 | Chrome・Edge を正式サポート。Safari は動作確認レベル |

## 7. 制約・前提条件

| 項目 | 内容 |
|------|------|
| 開発体制 | フリーランス1名（ryotasuyama）による開発 |
| 技術スタック | Next.js（App Router）/ Supabase（PostgreSQL + pgvector）/ Gemini API（`gemini-1.5-flash` を既定、要約・ドラフト生成用途）/ Gemini Embedding（`text-embedding-004`、RAG 用途）/ Vercel（**Pro 以上** を前提。毎分 Cron とナレッジ埋め込み生成に必要） |
| 音声文字起こし | Web Speech API を第一候補。精度不足の場合は Gemini API の音声入力を利用（外部音声文字起こし API への新規依存は追加しない） |
| ナレッジベース | 自前 RAG（pgvector による埋め込み検索）。NotebookLM Enterprise API は使用しない |
| メール送信 | アプリからの直接送信は MVP スコープ外。提案文のコピペ運用とする |
| 法規制 | 介護保険法・個人情報保護法に準拠した設計が必要。特に利用者情報の外部 API 送信には注意 |
| 初期ユーザー | 特定事業所のケアマネジャー。クローズドβとして運用開始 |

## 8. 用語集

| 用語 | 定義 |
|------|------|
| ケアプラン | 居宅サービス計画書。利用者の課題・目標・サービス内容を記載した計画書 |
| アセスメント | 利用者の状態・ニーズを把握するための情報収集・分析プロセス |
| モニタリング | ケアプランの実施状況を定期的に確認・評価すること |
| 要介護度 | 要支援1〜2、要介護1〜5の7段階で表される介護の必要度 |
| RAG | Retrieval-Augmented Generation。ナレッジベースを検索して回答精度を高める AI の仕組み |
| 個人ナレッジ | 各ケアマネジャーが登録する自分専用のメモ・ノウハウ |
| 共有ナレッジ | 管理者が登録し、事業所全員が参照できる制度資料・マニュアル類 |
| ドラフト | AI が生成するケアプランの原案。ケアマネが確認・編集して完成させる |
